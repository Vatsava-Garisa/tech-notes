
## Example Query
```
query {
	user(id: "1") {
		name
		posts {
			title
		}
	}
}
```
## Example Resolvers
```
const resolvers = {
	Query: {
		user: () => {...}
	},
	User: {
		posts: () => {...}
	}
}
```
---
## parent
- `parent` is the result returned by the previous resolver in the execution chain.
### Step-1
GraphQL executes:
```
user(id: "1")
```
Resolver:
```
user: async (_, args) => {
	return await UserModel.findById(arg.id);
}
```
The returned user object becomes the `parent` for all nested fields.
### Step-2
GraphQL resolves:
```
posts
```
Resolver:
```
User: {
	posts: async (parent) => {
		return await PostModel.find({ authorId: parent.id });
	}
}

/* 
	parent = previously resolved object. (user object)
*/
```
---
## args
- Arguments passed in the query.
- `args` is a plain object.
```
user(id: "1")
```
Resolver:
```
user: (_, args) => {
	console.log(args.id); // "1"
}
```
---
## context
- Per-request scoped.
- Created once per request.
- Shared across all resolvers.
You generally put:
- Current User
- Decoded JWT
- Data Loaders
- Request ID
- Logger
Equivalent to Express `req` but safer and controlled.
---
## info
Contains:
- Execution state
- Field selection tree
- Query AST
- Field name
- Path
Used for:
- Query complexity analysis
- Logging
- Performance tracing
- Advanced optimizations
---

| Param   | Meaning                  | Common Use        |
| ------- | ------------------------ | ----------------- |
| parent  | Previous resolver result | Nested resolution |
| args    | Client arguments         | Filters / input   |
| context | Request scope            | Auth, loaders     |
| info    | Execution metadata       | Advanced tooling  |
