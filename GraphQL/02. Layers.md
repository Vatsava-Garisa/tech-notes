
A GraphQL server has 3 essential layers:
1. Schema
2. Resolvers
3. Context

### Schema
- Strongly typed.
- Self-documenting.
- Client contract.
```
type User {
	id: ID!
	name: String!
	email: String!
}

input CreateUserInput {
	name: String!
	email: String!
}

type Query {
	getUser(id: ID!): User
}

type Mutation {
	createUser(input: CreateUserInput!): User!
}
```
### Resolvers
- Execution layer.
```
const resolvers = {
	Query: {
		getUser: async (_: any, args: { id: string }) => {
			return await UserModel.findById(args.id);
		}
	}
}

/* 
	(parent, args, context, info) 
	----------------------------------------------------
	|Param  |Meaning                 |Common Use       |
	----------------------------------------------------
	|parent |Previous resolver result|Nested resolution|
	|args   |Client arguments        |Filters / input  |
	|context|Request scope           |Auth, loaders    |
	|info   |Execution metadata      |Advanced tooling |
	----------------------------------------------------
*/
```
### Context (request scope)
- Equivalent of Express `req`.
- We inject: currentUser, DB Connections, Data Loaders, requestID etc.
---
```
	type    ->    Output
	input    ->    Arguments
	type Query    ->    Read Operations
	type Mutation    ->    Write Operations
	type Subscription    ->    Realtime
```

## `type` and `input`
- `type` = Return DTO with possible computed fields.
- `type` supports:
	- Resolvers
	- Interfaces
	- Unions
	- Field-level resolution
- `input` = TypeScript interface for request body.
- `input` is strictly:
	- JSON-like structured payload
	- No execution logic
	- No nested resolver hooks

```
type User {
	id: ID!
	name: String!
	email: String!
	password: String!
	createdAt: String!
}

/* Create User */
input CreateUserInput {
	name: String!
	email: String!
	password: String!
}

/* Update User */
input UpdateUserInput {
	name: String!
	email: String!
}

/* Get User */
type getUser {
	name: String!
	email: String!
}
```
---
## `interface`
Interfaces are about polymorphism in returned data.
They allow:
	Shared fields
	Runtime type resolution
	Union-like behavior
This required execution logic.
```
interface Node {
	id: ID!
}

type User implements Node {
	id: ID!
	name: String!
}

type Post implements Node {
	id: ID!
	name: String!
}
```
- `input` types:
	- Are validated before execution.
	- Cannot have runtime resolution.
	- Must be deterministic and static.
- If inputs supported interfaces:
	- The server would need runtime resolution.
	- Validation would become ambiguous.
	- Execution engine becomes more complex.
	- GraphQL developers intentionally avoided that.
