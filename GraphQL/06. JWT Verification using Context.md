
Usually in Express REST:
- Middleware runs before controller.
```
Request -> Middleware -> Route -> Controller
```
In GraphQL:
- Resolvers do not pass through Express route middleware per field.
- GraphQL has only one endpoint `/graphql`. So you cannot attach route-level middleware per operation.
- Context acts as request-level middleware.
```
Request -> Apollo -> Context Creation -> Resolver Execution
```
---
## What is Context?
When a GraphQL request comes in:
1. Apollo calls your `context()` function.
2. You inspect headers.
3. You decode JWT.
4. You attach `currentUser` to context.
5. Every resolver receives it.

**Example**
```
server.ts
---------

const server = new ApolloServer({
	typeDefs,
	resolvers
});

await startStandaloneServer(server, {
	context: async ({req} => {
		const token - req.headers.authorization?.split(" ")[1];
		
		let currentUser = null;
		
		if (token) {
			try {
				currentUser = jwt.verify(token. process.env.JWT_SECRET!);
			} catch (err) {
				throw new Error("Invalid token");
			}
		}
		
		return {
			currentUser
		};
	})
});


resolver.ts
-----------
Query: {
	me: (_, _, context, info) => {
		if (!context.currentUser) {
			throw new Error("Unauhorized");
		}
		
		return UserModel.findById(context.currentUser.id);
	}
}
```

- JWT is decoded once per request.
- No duplication across resolvers.
- Clean separation of concerns.
- Resolvers stay business-focused.

If we decode JWT in every resolver manually?
-----------------------------------------------------------------------------
### Performance Issue
- GraphQL executes field-level resolvers.
- One request could trigger many resolvers.
- If JWT verification runs inside each:
	- You decode the same token multiple times.
	- You waste CPU.
	- You increase latency.
```
query {
	user(id: "1") {
		posts {
			comments {
				author {
					name
				}
			}
		}
	}
}

// This triggers many resolvers.
```
### Violates DRY
- Auth logic becomes duplicated across resolvers.
- Change in auth logic -> refactor everywhere.
### Harder Testing
- If auth is centralized in context, you mock `context.currentUser` in tests.
- If auth is in every resolver, you must mock JWT verification repeatedly.
### Inconsistent Enforcement Risk
- Some resolver might forget to verify.
- Security hole.
